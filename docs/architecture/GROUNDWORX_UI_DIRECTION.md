# GroundworX UI Direction
# Canonical High-Performance Visual Language

---

## Status

This document is the canonical UI direction for GroundworX.

Reference sample:
- `C:\Dev\groundworx_high_performance_sample.html`

This replaces softer or more generic dashboard directions. GroundworX should
feel like a high-performance construction operations system with intelligence
built in.

---

## Core Design Statement

GroundworX should look like:

**a premium construction operations system with intelligence built in, not an
AI product pretending to be construction software.**

The visual language should communicate:
- speed
- control
- operational trust
- dense clarity
- human-led execution with machine-assisted insight

It should not communicate:
- consumer SaaS softness
- generic startup dashboard patterns
- theatrical “AI futurism”
- visual noise disguised as sophistication

---

## Mood

The dominant mood is:

**Signal Ops + Industrial Precision**

That means:
- dark graphite shell
- precise paneling
- mono labels for state and metadata
- strong sans-serif hierarchy for decision surfaces
- signal green used with discipline
- high-density information presentation

Secondary influence:

**Tactical Intelligence**

Use this selectively in:
- market intelligence
- overnight operations
- “what changed” or “what needs review” views

Do not let Tactical Intelligence overwhelm the whole product. The base shell
must remain controlled and credible.

---

## Color Philosophy

Signal green stays.

It is a **state color**, not a decorative color.

Use green for:
- active
- connected
- live
- selected
- recommended
- ready
- successful

Do not use green for:
- large paragraphs
- broad decorative fills
- every heading
- every border
- visual excitement for its own sake

Additional operational colors:
- amber = pending / attention / queued review
- red = blocked / failure / elevated risk
- steel blue = system / neutral intelligence / supporting metrics

White and muted gray must still carry most reading weight.

---

## Typography Direction

Primary fonts:
- `Barlow`
- `IBM Plex Mono`

Use them with role discipline:

- `Barlow 800/900`
  - major page titles
  - primary metrics
  - high-priority product headings

- `Barlow 600/700`
  - panel titles
  - section headings
  - strong table content

- `Barlow 400/500`
  - body copy
  - descriptive text
  - helper content

- `IBM Plex Mono 400/500`
  - labels
  - states
  - timestamps
  - metadata
  - chips
  - audit/provenance cues

Typography should do more work than color.

---

## Layout Philosophy

GroundworX should feel dense, not cramped.

Preferred shell pattern:
- top operational status bar
- left persistent navigation
- central primary work surface
- optional right intelligence / summary rail

The app should privilege:
- scan speed
- comparison
- operator confidence
- immediate next actions

Avoid over-centering and oversized “hero” layouts inside the product.

The marketing site may remain more atmospheric than the application shell.

---

## Motion Philosophy

Motion should be meaningful and sparse.

Allowed:
- live pulse indicators
- subtle hover transitions
- staged page reveal
- progress state changes
- panel expand/collapse

Avoid:
- constant scanline theatrics
- decorative motion on every surface
- pulsing panels
- animated glow for non-live elements

Motion should support state awareness, not style alone.

---

## Density Rules

GroundworX is an operations product. Density should reflect that.

Preferred behavior:
- compact headers
- clear but tight cards
- sticky controls where useful
- tables optimized for repeated use
- small metadata and labels
- strong spacing consistency

Design target:
- high signal-to-noise ratio
- low decorative waste
- clear urgency hierarchy

---

## Product Surface Priorities

The visual system should be introduced in this order:

1. settings
2. overnight operations / morning summary
3. market intelligence
4. project shell / overview
5. schedule / submittals / meetings / briefing

Reason:
- these surfaces fit the new language naturally
- they establish the design system without destabilizing core workflow logic

---

## Component Character

The system should feel:
- modular
- instrumented
- auditable
- exact

Preferred component types:
- metric cards
- status chips
- compact command bars
- audit rows
- queue cards
- provenance markers
- comparison panels
- summary rails
- high-density tables
- Glint recommendation panels

Each component should answer one of these questions quickly:
- What matters now?
- What changed?
- Is this safe?
- Is this blocked?
- What should I do next?
- Can I trust this output?

---

## Interaction Character

Buttons, filters, and controls should feel:
- crisp
- direct
- decisive

Avoid:
- soft rounded consumer controls
- oversized action buttons
- excess shadows
- playful motion

Prefer:
- compact heights
- mono uppercase labels for utility actions
- clear primary vs secondary action hierarchy

---

## What Makes It “High Performance”

GroundworX feels high performance when:
- green is reserved for true signal
- metrics and alerts are easy to scan
- tables feel engineered, not ornamental
- the shell feels stable under pressure
- audit and state information are visible
- overnight automation results read like operator outputs, not hidden logs

It stops feeling high performance when:
- visual effects become the main event
- too many accents compete
- every panel shouts
- the user has to hunt for what matters

---

## Hard Rules

1. Do not introduce generic blue SaaS styling into GroundworX.
2. Do not drift toward purple AI branding.
3. Do not use signal green as a blanket accent everywhere.
4. Do not design product screens like marketing pages.
5. Do not sacrifice density for trendiness.
6. Do not build “AI slop” card farms.
7. Do not over-round the interface.
8. Do not hide operational truth behind decorative polish.
